# ADR-0001: Use Five Strategy Iterations as a Public-Source Alignment Gate

## Status

Accepted

## Context

Naikaku is not meant to become a generic autonomous agent. The project direction is specific:

- Japan-built and Japanese-first.
- Useful for cautious Japanese enterprise automation and research workflows.
- Multi-role cabinet governance instead of unchecked single-agent execution.
- Bring-your-own-provider and future EMYSTI API gateway paths.
- Public-source contribution surface with commercial use controlled by 合同会社EMYSTI.
- Clear separation between dry-run, fixture, preview, and production evidence.

As the repository grows, it needs a repeatable way to check that new deployment, commercial, provider, runner, and documentation changes still support that direction.

## Decision

Add a five-iteration strategy report:

1. Japan-led enterprise positioning.
2. Role-based data governance.
3. Deployment and commercial readiness.
4. Professional open-source Git surface.
5. Verification and release evidence.

The report is generated by `npm run strategy:iterate` and is included in `npm run ci:open-source`. It emits JSON and Markdown artifacts under `output/strategy-iterations`.

## Consequences

### Positive

- The product thesis becomes machine-checkable instead of only prose.
- Commercial and deployment work must preserve licensing, security, and public-source boundaries.
- Contributors get a clearer definition of what "aligned with Naikaku" means.
- Release reviews can inspect one report before claiming the repository is ready.

### Negative

- The check is evidence-based and can miss subtle wording drift.
- New strategic pillars require updating both the domain report and documentation.
- Some evidence is necessarily documentation-level until runtime enforcement is implemented.

### Neutral

- The report complements `deployment:check`, `open-source:mvp-check`, and release verification. It does not replace runtime security checks, provider tests, runner evidence, or production-mode verification.

## Alternatives Considered

**README-only guidance**

- Rejected because it is easy to drift and hard to enforce in CI.

**Single deployment readiness check**

- Rejected because deployment readiness does not cover Japanese positioning, role data governance, open-source contribution surface, or release-evidence honesty.

**Full production compliance framework now**

- Rejected as premature. The current repository is a public-source developer preview and should keep operational complexity proportional.
